## 🎯 H.264 是什么？

### 官方定义

**H.264**，也叫 **MPEG-4 Part 10** 或 **AVC (Advanced Video Coding)**，是一种视频压缩标准。

### 简单理解

**H.264 就像是一个超级压缩器**，可以把很大的视频文件压缩得很小，同时保持较好的画质。

### 生活比喻

```
📦 原始视频（未压缩）
├── 大小：1 GB
└── 问题：太大了，无法分享

🗜️ H.264 压缩后
├── 大小：50 MB（压缩了 20 倍！）
├── 画质：依然清晰
└── 优点：容易存储和传输
```

### H.264 的特点

| 特点 | 说明 |
|-----|------|
| 🎯 **高压缩率** | 比 MPEG-2 节省 50% 以上带宽 |
| 🌍 **广泛应用** | YouTube、Netflix、微信视频等都用它 |
| 📱 **设备兼容** | 所有手机、电脑都支持 |
| ⚡ **适应性强** | 支持从手机到 4K 的各种分辨率 |

## 🎯 H.264 的核心概念

### 1. 编解码器（Codec）

```
编码器（Encoder）：
原始视频帧 → H.264 压缩数据

解码器（Decoder）：
H.264 压缩数据 → 原始视频帧
```

### 2. 比特流（Bitstream）

H.264 编码后的数据就是**比特流**，就像一条数据河流。

```
比特流 = H.264 编码后的所有数据
       = 头部信息 + 帧数据 + 尾部信息
```

### 3. 序列（GOP - Group of Pictures）

**GOP** 就是一组连续的帧，通常以 I 帧开始。

```
GOP 示例（长度=4）：
I(关键帧) → P → B → P → I → P → B → P → ...

每个 GOP 独立编码，互不影响
```

## 🎯 视频压缩原理

### 为什么视频可以压缩？

#### 空间冗余（Spatial Redundancy）

```
同一帧图像中：
相邻像素通常是相似的

例如：
aaaaaabbbbbbcccccc
可以压缩为：6a 6b 6c（游程编码）
```

#### 时间冗余（Temporal Redundancy）

```
连续两帧之间：
只有很小的变化

第1帧：🎬🎬🎬🎬🎬🎬🎬🎬
第2帧：🎬🎬🎬🎬🎬🎬🎬🎬 （几乎一样！）
     ↑ 只有这个小球位置变了

不需要传输整个第2帧！
只需要传输"变化"的部分
```

### H.264 的压缩技术

| 技术 | 说明 | 压缩效果 |
|-----|------|---------|
| **帧内预测** | 利用同一帧内的相似性 | 🟢 中等 |
| **帧间预测** | 利用帧与帧之间的相似性 | 🟢🟢 很高 |
| **变换编码** | 将图像从空间域转到频域 | 🟢🟢 很高 |
| **量化** | 丢弃不重要的信息 | 🟢🟢🟢 最高 |
| **熵编码** | 高效编码（如 CABAC） | 🟢 中等 |

## 🎯 H.264 编码结构

### 分层架构

```
┌─────────────────────────────────────────────┐
│         H.264 比特流（Bitstream）            │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│  序列层（Sequence Layer）                     │
│  ├── SPS (序列参数集)                        │
│  └── PPS (图像参数集)                        │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│  图像层（Picture Layer）                     │
│  ├── I 帧                                   │
│  ├── P 帧                                   │
│  └── B 帧                                   │
└─────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────┐
│  宏块层（Macroblock Layer）                  │
│  ├── 16×16 像素块                           │
│  ├── 预测模式                              │
│  └── 残差数据                              │
└─────────────────────────────────────────────┘
```

### 1. SPS（序列参数集）

**SPS** 包含整个视频序列的全局信息：

```json
{
  "profile_idc": "High Profile",      // 编码档次
  "level_idc": "Level 4.0",          // 编码级别
  "pic_width_in_mbs": 80,             // 帧宽度（宏块数）
  "pic_height_in_map_units": 45,      // 帧高度
  "frame_mbs_only_flag": 1,           // 逐行扫描
  "vui_parameters": {                 // 视频可用性信息
    "timing_info": {
      "num_units_in_tick": 1001,
      "time_scale": 60000
    }
  }
}
```

### 2. PPS（图像参数集）

**PPS** 包含单个图像的编码参数：

```json
{
  "pic_parameter_set_id": 0,
  "seq_parameter_set_id": 0,
  "entropy_coding_mode_flag": 1,      // CABAC 熵编码
  "num_ref_idx_l0_default_active": 1, // 前向参考帧数
  "num_ref_idx_l1_default_active": 1, // 后向参考帧数
  "weighted_pred_flag": 0,            // 加权预测
  "deblocking_filter_control": 1      // 去块滤波
}
```

### 3. 实际例子

```
完整 H.264 文件：

[SPS][PPS][I帧][P帧][B帧][B帧][P帧][I帧][P帧]...

每个 GOP：
┌──────────────────────────────────────┐
│ [SPS][PPS] ← 只在 GOP 开始时出现      │
│ [I帧] ← GOP 的第一帧，关键帧          │
│ [P帧] ← 向前预测帧                    │
│ [B帧] ← 双向预测帧                    │
│ [P帧] ← 向前预测帧                    │
└──────────────────────────────────────┘
```

## 🎯 帧类型详解

### 三种帧类型

| 帧类型 | 英文名 | 说明 | 大小（相对） | 依赖 |
|-------|--------|------|-------------|------|
| **I 帧** | Intra-coded | 完整编码的帧 | 🔴 最大 | 无 |
| **P 帧** | Predictive | 前向预测帧 | 🟡 中等 | I 帧或 P 帧 |
| **B 帧** | Bi-directional | 双向预测帧 | 🟢 最小 | I 帧和 P 帧 |

### 1. I 帧（关键帧）

**I 帧是自包含的**，完整编码了整个图像，不需要参考其他帧。

```
I 帧编码过程：
┌─────────────────┐
│   原始图像      │
│  [🎬🎬🎬🎬🎬🎬] │
└────────┬────────┘
         ↓
┌─────────────────┐
│   帧内预测      │ ← 利用同一帧内的相似性
│  [预测块] [残差] │
└────────┬────────┘
         ↓
┌─────────────────┐
│   变换 + 量化   │ ← DCT 变换，丢弃不重要信息
│   [压缩数据]    │
└────────┬────────┘
         ↓
┌─────────────────┐
│   熵编码        │ ← CABAC/CAVLC
│   [I帧比特流]   │
└─────────────────┘
```

### 2. P 帧（预测帧）

**P 帧只编码与前面参考帧的差异**。

```
P 帧编码过程：
┌─────────────────┐     ┌─────────────────┐
│   当前帧        │     │   参考帧（I/P） │
│  [🎬  🎬🎬🎬🎬🎬] │     │  [🎬🎬🎬🎬🎬🎬🎬] │
└────────┬────────┘     └────────┬────────┘
         │                         │
         └────────┬────────────────┘
                  ↓
         ┌─────────────────┐
         │   运动估计/补偿  │ ← 找到运动方向
         │  [运动矢量]      │
         └────────┬────────┘
                  ↓
         ┌─────────────────┐
         │   预测残差      │ ← 只有差异部分
         │  [🎬-🎬 = 0]   │
         └────────┬────────┘
                  ↓
         ┌─────────────────┐
         │   编码          │
         │   [P帧比特流]   │
         └─────────────────┘
```

### 3. B 帧（双向预测帧）

**B 帧同时参考前面和后面的帧**。

```
B 帧编码过程：
┌─────────────────┐
│   当前帧        │
│  [🎬🎬🎬🎬🎬🎬] │
└────────┬────────┘
         │
    ┌────┴────┐
    ↓         ↓
┌───────┐  ┌───────┐
│前向参考│  │后向参考│
│ 🎬🎬🎬 │  │ 🎬🎬🎬 │
└───┬───┘  └───┬───┘
    │          │
    └────┬─────┘
         ↓
   ┌───────────┐
   │ 双向预测  │ ← 取两个参考的平均
   │ [平均值]  │
   └─────┬─────┘
         ↓
   ┌───────────┐
   │   编码    │
   │ [B帧比特流]│
   └───────────┘
```

### 帧的实际例子

```
视频序列（30fps，1秒）：
时间：0.00  0.03  0.06  0.09  0.12  0.15 ...
帧：  I     P     B     B     P     B     B     P     I ...

GOP 长度 = 4 的示例：
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ I  │ P  │ B  │ B  │ I  │ P  │ B  │ B  │
└────┴────┴────┴────┴────┴────┴────┴────┘
 0    1    2    3    4    5    6    7  (帧号)

播放顺序：I(0) → P(1) → B(2) → B(3) → I(4) → P(5) → B(6) → B(7)
存储顺序：I(0) → P(1) → B(2) → B(3) → I(4) → P(5) → B(6) → B(7)
```

## 🎯 H.264 解码流程

### 解码器工作流程

```
         ┌─────────────────────┐
         │   H.264 比特流      │
         │ [SPS][PPS][I][P][B] │
         └──────────┬──────────┘
                    ↓
         ┌─────────────────────┐
         │   熵解码           │ ← CABAC/CAVLC
         │  [压缩数据 → 系数]  │
         └──────────┬──────────┘
                    ↓
         ┌─────────────────────┐
         │   反量化           │ ← 恢复量化后的系数
         │   反变换           │ ← IDCT 逆变换
         └──────────┬──────────┘
                    ↓
         ┌─────────────────────┐
         │   运动补偿         │ ← 利用运动矢量重建
         │  [参考帧 → 当前帧] │
         └──────────┬──────────┘
                    ↓
         ┌─────────────────────┐
         │   解码图像          │
         │  [完整的视频帧]     │
         └─────────────────────┘
```

### Android H.264 解码示例

```kotlin
/**
 * H.264 解码器
 *
 * 完整流程：
 * 1. 创建解码器
 * 2. 配置解码器
 * 3. 循环输入压缩数据
 * 4. 循环输出原始帧
 * 5. 渲染到 Surface
 */
class H264Decoder {

    private var decoder: MediaCodec? = null
    private var surface: Surface? = null

    /**
     * 初始化解码器
     *
     * @param mimeType 必须是 "video/avc"（H.264）
     * @param surface 用于渲染解码后的图像
     */
    fun initialize(mimeType: String = "video/avc", surface: Surface) {
        this.surface = surface

        // 步骤 1：创建解码器
        decoder = MediaCodec.createDecoderByType(mimeType)

        // 步骤 2：配置解码器
        // format 包含视频的宽高、帧率等信息
        decoder!!.configure(format, surface, null, 0)

        // 步骤 3：启动解码器
        decoder!!.start()
    }

    /**
     * 解码一个压缩的 H.264 数据块
     *
     * @param inputBuffer 压缩的 H.264 数据（如一帧）
     * @param presentationTimeUs 显示时间戳（微秒）
     */
    fun decode(inputBuffer: ByteBuffer, presentationTimeUs: Long) {
        // ========== 输入阶段 ==========

        // 获取输入缓冲区的索引
        val inputBufferIndex = decoder!!.dequeueInputBuffer(10000) // 10ms 超时

        if (inputBufferIndex >= 0) {
            // 获取实际的缓冲区
            val decoderInputBuffer = decoder!!.getInputBuffer(inputBufferIndex)

            if (decoderInputBuffer != null) {
                // 写入压缩数据
                decoderInputBuffer.clear()
                decoderInputBuffer.put(inputBuffer)

                // 通知解码器有数据输入
                decoder!!.queueInputBuffer(
                    inputBufferIndex,
                    0,                          // 偏移量
                    inputBuffer.remaining(),     // 数据大小
                    presentationTimeUs,          // 时间戳
                    0                            // 标志
                )
            }
        }

        // ========== 输出阶段 ==========

        val bufferInfo = MediaCodec.BufferInfo()
        var outputBufferIndex: Int

        do {
            outputBufferIndex = decoder!!.dequeueOutputBuffer(bufferInfo, 10000)

            when (outputBufferIndex) {
                // 没有可用的输出
                MediaCodec.INFO_TRY_AGAIN_LATER -> {
                    // 稍后再试
                }

                // 输出格式改变了
                MediaCodec.INFO_OUTPUT_FORMAT_CHANGED -> {
                    val newFormat = decoder!!.outputFormat
                    // 可以获取新的视频参数
                    val width = newFormat.getInteger(MediaFormat.KEY_WIDTH)
                    val height = newFormat.getInteger(MediaFormat.KEY_HEIGHT)
                }

                // 有可用的解码帧
                else -> {
                    if (outputBufferIndex >= 0) {
                        // presentationTimeUs 是这帧的显示时间
                        // duration 是帧的时长（通常不用）

                        // 渲染到 Surface
                        //第二个参数 true 表示渲染到 Surface
                        decoder!!.releaseOutputBuffer(outputBufferIndex, true)
                    }
                }
            }
        } while (outputBufferIndex >= 0)
    }

    /**
     * 释放资源
     */
    fun release() {
        decoder?.stop()
        decoder?.release()
        decoder = null
    }
}
```

## 🎯 H.264 编码流程

### 编码器工作流程

```
┌─────────────────────────────────────────────┐
│              原始视频帧                       │
│            [YUV 格式]                       │
└─────────────────┬───────────────────────────┘
                  ↓
┌─────────────────────────────────────────────┐
│              帧类型预测                       │
│         [I 帧？P 帧？B 帧？]                 │
└─────────────────┬───────────────────────────┘
                  ↓
         ┌────────┴────────┐
         ↓                ↓
    ┌─────────┐      ┌─────────┐
    │ 帧内预测 │      │ 帧间预测 │
    │ (I 帧)  │      │ (P/B)   │
    └────┬────┘      └────┬────┘
         │                │
         └────────┬───────┘
                  ↓
         ┌─────────────────┐
         │    残差计算     │ ← 预测值与原值的差异
         └────────┬────────┘
                  ↓
         ┌─────────────────┐
         │   DCT 变换      │ ← 空间域 → 频域
         └────────┬────────┘
                  ↓
         ┌─────────────────┐
         │   量化 (Q)      │ ← 丢弃不重要的高频信息
         └────────┬────────┘
                  ↓
         ┌─────────────────┐
         │   熵编码        │ ← CABAC/CAVLC
         └────────┬────────┘
                  ↓
         ┌─────────────────┐
         │   H.264 比特流  │
         │  [压缩数据]     │
         └─────────────────┘
```

### Android H.264 编码示例

```kotlin
/**
 * H.264 编码器
 */
class H264Encoder {

    private var encoder: MediaCodec? = null

    /**
     * 初始化编码器
     *
     * @param width 视频宽度
     * @param height 视频高度
     * @param bitRate 比特率（越高越清晰）
     * @param frameRate 帧率
     */
    fun initialize(
        width: Int = 1280,
        height: Int = 720,
        bitRate: Int = 5_000_000,     // 5 Mbps
        frameRate: Int = 30            // 30 fps
    ) {
        // 步骤 1：创建 H.264 编码器
        encoder = MediaCodec.createEncoderByType("video/avc")

        // 步骤 2：创建格式
        val format = MediaFormat.createVideoFormat("video/avc", width, height).apply {
            // 比特率：每秒 5M 位
            setInteger(MediaFormat.KEY_BIT_RATE, bitRate)

            // 帧率
            setInteger(MediaFormat.KEY_FRAME_RATE, frameRate)

            // 关键帧间隔：2 秒
            setInteger(MediaFormat.KEY_I_FRAME_INTERVAL, 2)

            // 颜色格式（YUV）
            setInteger(
                MediaFormat.KEY_COLOR_FORMAT,
                MediaCodecInfo.CodecCapabilities.COLOR_FormatSurface
            )
        }

        // 步骤 3：配置编码器
        encoder!!.configure(format, null, null, MediaCodec.CONFIGURE_FLAG_ENCODE)

        // 步骤 4：启动编码器
        encoder!!.start()
    }

    /**
     * 编码一帧
     *
     * @param presentationTimeUs 时间戳
     * @return 编码后的 H.264 数据
     */
    fun encodeFrame(presentationTimeUs: Long): ByteArray? {
        // ========== 输入阶段 ==========

        val inputBufferIndex = encoder!!.dequeueInputBuffer(10000)

        if (inputBufferIndex >= 0) {
            // 获取输入缓冲区（通常是 Surface）
            val inputBuffer = encoder!!.getInputBuffer(inputBufferIndex)

            if (inputBuffer != null) {
                // 在这里填充原始视频数据（YUV）
                // inputBuffer.put(yuvData)

                // 通知编码器数据已输入
                encoder!!.queueInputBuffer(
                    inputBufferIndex,
                    0,
                    inputBuffer.remaining(),
                    presentationTimeUs,
                    0
                )
            }
        }

        // ========== 输出阶段 ==========

        val bufferInfo = MediaCodec.BufferInfo()
        val outputBufferIndex = encoder!!.dequeueOutputBuffer(bufferInfo, 10000)

        return when (outputBufferIndex) {
            MediaCodec.INFO_TRY_AGAIN_LATER -> null

            MediaCodec.INFO_OUTPUT_FORMAT_CHANGED -> {
                // 格式改变了（第一次会触发）
                val newFormat = encoder!!.outputFormat
                null
            }

            else -> {
                if (outputBufferIndex >= 0) {
                    val outputBuffer = encoder!!.getOutputBuffer(outputBufferIndex)
                    val encodedData = outputBuffer?.let {
                        val data = ByteArray(it.remaining())
                        it.get(data)
                        data
                    }

                    // 重要：判断是否是关键帧
                    val isKeyFrame = (bufferInfo.flags and MediaCodec.BUFFER_FLAG_KEY_FRAME) != 0

                    if (isKeyFrame) {
                        // 关键帧，需要在前面加 SPS/PPS
                        // 通常在 INFO_OUTPUT_FORMAT_CHANGED 时获取
                    }

                    encoder!!.releaseOutputBuffer(outputBufferIndex, false)

                    bufferInfo.size.takeIf { it > 0 }?.let { size ->
                        // 返回编码后的数据
                        ByteArray(size).also { arr ->
                            outputBuffer?.let {
                                it.position(bufferInfo.offset)
                                it.get(arr, 0, size)
                            }
                        }
                    }
                } else null
            }
        }
    }

    /**
     * 释放资源
     */
    fun release() {
        encoder?.stop()
        encoder?.release()
        encoder = null
    }
}
```

## 🎯 H.264 关键技术

### 1. 运动估计和运动补偿（ME/MC）

**运动估计**就是找到当前块在参考帧中的最佳匹配位置。

```
当前帧的某个 16×16 宏块：

┌────────────────────────┐
│                        │
│      [████████]        │ ← 要编码的宏块
│                        │
└────────────────────────┘

参考帧：
┌────────────────────────┐
│      [████████]        │ ← 在这里找到了相似的块
│                        │   运动矢量 = (+5, -3)
└────────────────────────┘

只需要编码：
1. 运动矢量：(+5, -3)
2. 预测误差：当前块 - 预测块
```

### 2. 帧内预测

**帧内预测**利用同一帧内相邻像素来预测当前块。

```
4×4 块的帧内预测：

┌────────────────┐
│ a b c d        │← 上面一行是已编码的像素
│ ○ ○ ○ ○        │← ○ 是要预测的像素
│ ○ ○ ○ ○        │
│ ○ ○ ○ ○        │
└────────────────┘

a, b, c, d 是已编码的像素
○ 是当前要编码的像素

预测模式：
- 模式 0（垂直）：○ = a（下方像素 = 上方像素）
- 模式 1（水平）：○ = b（右侧像素 = 左侧像素）
- 模式 2（DC）：○ = 所有像素的平均值
```

### 3. 整数 DCT 变换

H.264 使用整数 DCT 变换，比浮点 DCT 更快更稳定。

```
DCT 变换的作用：
┌────────────────────────────────┐
│        空间域（图像）          │
│  [像素值：100, 80, 60, 40...]  │
└────────────────┬───────────────┘
                 ↓ DCT
┌────────────────────────────────┐
│        频域（系数）            │
│  [DC系数, AC系数1, AC系数2...] │
└────────────────┬───────────────┘
                 ↓ 量化
┌────────────────────────────────┐
│   [重要系数保留，不重要的置零]  │
│   [压缩效果明显！]              │
└────────────────────────────────┘
```

### 4. 去块滤波器（Deblocking Filter）

每个宏块边界可能会有"方块效应"，去块滤波器可以平滑这些边界。

```
没有去块滤波：
┌────┬────┬────┐
│████│████│████│ ← 明显的方块边界
├────┼────┼────┤
│████│████│████│
└────┴────┴────┘

有去块滤波后：
┌─────────────────┐
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓ │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓ │ ← 平滑的边界
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓ │
└─────────────────┘
```

## 🎯 实际代码示例

### 示例：从 MP4 提取 H.264 数据

```kotlin
/**
 * 从 MP4 文件提取 H.264 视频流
 */
class H264Extractor {

    private val extractor = MediaExtractor()

    /**
     * 设置数据源
     */
    fun setDataSource(path: String) {
        extractor.setDataSource(path)
    }

    /**
     * 找到 H.264 视频轨道
     */
    fun findH264Track(): Int {
        for (i in 0 until extractor.trackCount) {
            val format = extractor.getTrackFormat(i)
            val mime = format.getString(MediaFormat.KEY_MIME)

            if (mime?.startsWith("video/") == true && mime.contains("avc")) {
                extractor.selectTrack(i)
                return i
            }
        }
        return -1
    }

    /**
     * 读取一帧 H.264 数据
     *
     * @return 包含 start code (0x00000001) 的 H.264 NAL 单元
     */
    fun readFrame(): ByteArray? {
        val buffer = ByteBuffer.allocate(1024 * 1024)
        val size = extractor.readSampleData(buffer, 0)

        if (size < 0) {
            return null
        }

        val data = ByteArray(size)
        buffer.rewind()
        buffer.get(data, 0, size)

        // 添加 H.264 start code
        return byteArrayOf(0x00, 0x00, 0x00, 0x01) + data
    }

    /**
     * 跳到指定时间
     */
    fun seekTo(timeUs: Long) {
        extractor.seekTo(timeUs, MediaExtractor.SEEK_TO_CLOSEST_SYNC)
    }

    /**
     * 获取当前时间
     */
    fun getCurrentTime(): Long {
        return extractor.sampleTime
    }

    /**
     * 释放资源
     */
    fun release() {
        extractor.release()
    }
}
```

### 示例：H.264 NAL 单元解析

```kotlin
/**
 * H.264 NAL 单元类型
 */
enum class NaluType(val value: Int, val name: String) {
    SLICE(1, "非关键帧切片"),
    IDR(5, "关键帧切片"),
    SPS(7, "序列参数集"),
    PPS(8, "图像参数集"),
    SEI(6, "补充增强信息"),
    AUD(9, "访问单元分隔符")
}

/**
 * 解析 H.264 NAL 单元类型
 */
fun parseNaluType(data: ByteArray): NaluType {
    // NAL 头字节：0x00000001 之后的第一字节
    val nalType = data[4].toInt() and 0x1F

    return when (nalType) {
        1 -> NaluType.SLICE
        5 -> NaluType.IDR
        7 -> NaluType.SPS
        8 -> NaluType.PPS
        6 -> NaluType.SEI
        9 -> NaluType.AUD
        else -> NaluType.SLICE
    }
}

/**
 * 检查是否是关键帧
 */
fun isKeyFrame(data: ByteArray): Boolean {
    return parseNaluType(data) == NaluType.IDR
}
```

## 🎯 常见问题解答

### Q1: H.264 的 profile 是什么？

**A**: H.264 有三个主要 profile：

| Profile | 说明 | 特点 | 应用场景 |
|---------|------|------|---------|
| **Baseline** | 基础 | 无 B 帧，无 CABAC | 早期设备、视频通话 |
| **Main** | 主流 | 有 B 帧，CAVLC | 广播、DVD |
| **High** | 高级 | B 帧 + CABAC + 8x8 变换 | 高清电视、蓝光 |

### Q2: H.264 的 level 是什么？

**A**: Level 表示编解码器的能力级别：

| Level | 最大分辨率 | 最大帧率 | 最大比特率 |
|-------|----------|----------|-----------|
| 1 | 320×240 | 15 fps | 64 kbps |
| 1.1 | 352×288 | 30 fps | 192 kbps |
| 2 | 720×480 | 30 fps | 2 Mbps |
| 2.1 | 1280×720 | 30 fps | 4 Mbps |
| 3 | 1920×1080 | 30 fps | 10 Mbps |
| 4 | 1920×1080 | 60 fps | 20 Mbps |

### Q3: 为什么视频快进时会出现花屏？

**A**: 因为快进跳到了非关键帧位置。

```
正常播放：
I → P → B → B → P → B → B → I
↓   ↓   ↓   ↓   ↓   ↓   ↓   ↓
正常 正常 正常 正常 正常 正常 正常 正常

快进到 P4（非关键帧）：
I → P → P → P → P → P → P → I
     ↑  ↑
     这里开始播放会花屏！
     因为 P4 需要参考 I，但跳过了 I

解决方法：快进必须跳到关键帧（I 帧）
```

### Q4: 什么是码率控制？

**A**: 码率控制决定每个帧用多少数据表示。

| 模式 | 说明 | 特点 |
|-----|------|------|
| **CBR** | 恒定码率 | 带宽稳定，质量波动 |
| **VBR** | 变码率 | 质量稳定，带宽波动 |
| **ABR** | 平均码率 | 平衡方案 |

### Q5: H.264 如何实现超低延迟？

**A**: 使用以下技术：

1. **降低 GOP 长度**：减少关键帧间隔
2. **禁用 B 帧**：减少编码延迟
3. **使用 CAVLC**：比 CABAC 更快
4. **配置 `profile-level-id`**：优化编码参数

### Q6: 为什么 H.264 视频有绿边或花屏？

**A**: 常见原因：

1. **码率不够**：画面复杂时质量下降
2. **GOP 太长**：关键帧间隔太大
3. **设备性能不足**：编码/解码速度跟不上
4. **Reference 帧错误**：B 帧依赖的参考帧有问题

## 🎯 总结

### H.264 核心要点

```
1️⃣ H.264 是视频压缩标准
   └── 广泛用于所有视频应用

2️⃣ 三种帧类型
   ├── I 帧：完整图像，关键帧
   ├── P 帧：前向预测，依赖 I/P
   └── B 帧：双向预测，依赖 I 和 P

3️⃣ GOP（图片组）
   ├── 由 I 帧开始
   ├── 包含 I、P、B 帧
   └── 决定随机访问能力

4️⃣ Android 支持
   ├── MediaCodec API
   ├── MIME: "video/avc"
   └── 需要处理 SPS/PPS

5️⃣ 关键帧很重要
   ├── 视频剪辑必须从关键帧开始
   ├── 快进必须跳到关键帧
   └── 关键帧越大，但便于随机访问
```
